3 Approaches: Hackathon Mode

How should hackathon login + mode be wired into the app?

A

Separate login button → isHackathon flag in AuthContext

Add a "Hackathon Login" button on the landing page (email + password). After signInWithPassword, check hackathon_team_members for the user. If found, set isHackathon: true in AuthContext. RootNavigator routes hackathon users to /(hackathon)/ tabs instead of /(tabs)/.

Pros

  • Clean separation — one flag drives all routing
  • Follows existing guest/session pattern in auth.tsx
  • No new context files needed

Cons

  • AuthContext grows slightly

⭐ Recommended

B

Separate HackathonAuthContext

Keep normal auth completely untouched. Create a parallel HackathonAuthProvider with its own email/password state. The landing page shows both buttons; hackathon users get their own context tree.

Pros

  • Zero risk of affecting existing auth

Cons

  • Two auth contexts = complex nesting
  • Duplicates session management logic
  • Harder to maintain
C

Hackathon as a separate Expo route group with own layout

Add a /(hackathon-login)/ stack that handles its own auth entirely. No changes to lib/auth.tsx at all. The landing page navigates to a hackathon-specific login screen that manages its own session.

Pros

  • Fully isolated — can't break normal users

Cons

  • Duplicate Supabase session handling
  • Tricky with expo-router's single root layout
  • Two sessions could conflict
